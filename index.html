1 · Amazon RDS — a managed database

RDS is usually preferred over installing a database manually on an EC2 instance. The simplest way to frame it:

  • EC2 + Database = you are the database administrator (DBA).
  • RDS = AWS becomes the DBA for most operational tasks.

Supported database engines

RDS runs six relational engines: MySQL, PostgreSQL, MariaDB, Oracle, Microsoft SQL Server and Amazon Aurora. It does not run NoSQL databases such as MongoDB (that would be DynamoDB or DocumentDB instead).

RDS is a managed service

If you install MySQL/PostgreSQL on EC2 yourself, you must launch EC2, install the engine, configure storage, configure backups, and keep it all running. With RDS you just choose the engine, instance size and storage size and click Create Database. AWS then automatically installs the database, configures storage, starts the DB service, and creates the connection endpoint.

Automated OS patching

Every database server runs on an operating system (Amazon Linux, Ubuntu, Windows). Security vulnerabilities are found regularly, and RDS applies these OS and engine patches for you instead of you doing manual maintenance.

Storage backed by EBS

RDS storage uses Amazon EBS behind the scenes, which is durable, persistent, supports snapshots, and is high-performance. Common storage types include General Purpose SSD (gp3) and Provisioned IOPS SSD (io1/io2).

Horizontal scaling

To handle more readers you add read replicas — scaling out from one database, to several replicas, to many. More replicas means more users can read data simultaneously.

1 Database↓ 3 Read Replicas↓ 10 Read Replicas

An RDS database can have up to 5 read replicas (the classic limit; AWS has since raised this to 15 for several engines). Aurora, by contrast, supports up to 15 — see the Aurora section.

RDS encryption

Encryption must be decided up front. You cannot simply toggle encryption on a running, already-created instance. To encrypt an existing un-encrypted instance you: take a snapshot → copy the snapshot with “Enable encryption” ticked (choosing a KMS key) → restore a new instance from the encrypted snapshot.

Unencrypted DB↓ snapshot Snapshot↓ copy + Enable encryption (KMS) Encrypted snapshot↓ restore New encrypted DB
🔒

A read replica's encryption must match its source. If the primary is un-encrypted, its read replicas cannot be encrypted — you'd first have to encrypt the primary (via the snapshot method above).

IAM Database Authentication

Instead of creating a separate database username/password for every developer, you can enable IAM Database Authentication. Users then connect to the database using their existing IAM identities via a short-lived auth token — no per-user DB password to manage.

⚠️

IAM Database Authentication is supported only for the MySQL and PostgreSQL engines. Oracle does NOT support it. (Amazon Cognito is for app end-user sign-in, not database access.)

2 · Read Replicas

A read replica is a read-only copy of a primary database that receives data through replication and is used mainly for read (SELECT) operations.

The problem

Suppose 1,000 users are reading data and 50 are writing. A single database becomes overloaded.

The solution

Primary DB (Master) ↓ replication Replica 1 Replica 2

Writes (INSERT, UPDATE, DELETE) go to the primary database. Reads (SELECT) are spread across the replicas. So in the diagram, M = Master (primary) and R = Read Replica.

Why we need read replicas

They increase read performance, reduce load on the main database, and let you scale applications that have many users reading data.

Asynchronous replication

RDS read replicas use async replication: the primary updates first, and the replica updates a moment later. The primary does not wait for replicas before responding. That short delay — which can be 100 ms, 500 ms, or even a couple of seconds depending on workload — is called replication lag.

🚫

Read replicas are read-only. You cannot run INSERT/UPDATE/DELETE on them — only SELECT.

🔗

Endpoints differ. Each read replica gets its own new endpoint, so using one means updating your app's connection string to point reads at it. Multi-AZ, on the other hand, keeps the same endpoint after failover — so enabling Multi-AZ requires no connection-string change.

Where replicas can live

PlacementBenefit
Same AZ (e.g. Mumbai AZ-1: primary + replica)Low latency
Cross-AZ (AZ-1 primary → AZ-2 replica)Better fault tolerance
Cross-Region (replicate to another region)Disaster recovery at the regional level + serve reads closer to users

3 · Multi-AZ & the standby

Question asked

In a Multi-AZ setup the standby RDS is created via replication from the primary — can we do read and write operations on the standby?

❌

No. The Multi-AZ standby serves neither reads nor writes. It exists purely for high availability.

How Multi-AZ writes work (synchronous)

Multi-AZ uses synchronous replication. When the application writes, e.g. INSERT INTO Orders VALUES(101,'Laptop'):

Application writes↓ Primary → Saved↓ sync Standby → Saved↓ Standby confirms “Data successfully written”↓ Only then AWS returns Success to the app

Because AWS waits for the standby to confirm before reporting success, both databases always contain the same data up to the last committed transaction.

Failover & the DNS switch

If the primary crashes, AWS promotes the standby, updates the DNS endpoint to point to the new primary, and the application reconnects. The endpoint name stays the same; only what it points to changes.

Read Replica vs Multi-AZ standby

These are often confused because both involve replication, but they solve different problems.

FeatureRead ReplicaMulti-AZ Standby
PurposeScale readsHigh availability
Serves readsYesNo
Serves writesNoNo
ReplicationAsynchronousSynchronous
Automatic failoverNo (RDS)Yes
Improves performanceYesNo

4 · Amazon Aurora

Aurora is AWS's cloud-native engine, compatible with MySQL and PostgreSQL, so the same SQL works unchanged — e.g. SELECT * FROM users; runs exactly the same.

Architecture

Application↓ Writer Endpoint Writer Node (Master)↓ shared storage Reader 1 Reader 2 Reader 3

An Aurora cluster has 1 writer (master) and up to 15 readers (replicas), and crucially all of them share the same storage layer. The storage automatically keeps 6 copies of the data across 3 Availability Zones, which is why large platforms (Netflix-scale e-commerce, banking systems) can store massive datasets durably.

The two endpoints

EndpointRole
Writer EndpointAlways points to the current writer node; only the writer can execute writes.
Reader Endpoint
mydb.cluster-ro-xyz.amazonaws.com		Load-balances read requests across all replicas (connection load balancing).

Why Aurora is faster than normal RDS

In normal RDS the MySQL engine writes data directly to attached EBS storage. Aurora instead pushes writes to a distributed, purpose-built storage layer shared by all nodes — removing much of the traditional write overhead.

Why Aurora failover is faster than RDS Multi-AZ

RDS Multi-AZ has a separate standby with its own storage, so failover means promote standby → update DNS → reconnect, which takes time. In Aurora every node already shares one storage layer, so if the writer crashes Aurora simply promotes the healthiest reader (e.g. Reader 1 → New Writer) and updates the writer endpoint. Applications keep working with minimal delay.

The 6-copy write — why not wait for all 6?

Aurora keeps 6 copies but does not require all six to confirm a write. If it waited for every copy and, say, Node 5 was slow and Node 6 had failed, the whole application would slow down. Instead Aurora accepts the write once 4 of 6 confirmations arrive, which provides both speed and fault tolerance. Reads only need 3 of 6 copies.

🌐

Aurora replicas use the shared storage, so they typically have far less replication lag than normal RDS async replicas. Cross-region Aurora replication provides regional disaster recovery.

5 · RDS Proxy

Question asked

Explain RDS Proxy in detail.

RDS Proxy is a fully managed connection pool that sits between the application and the database. The application talks to the proxy; the proxy talks to the database.

Application↓ RDS Proxy (pool)↓ RDS / Aurora DB

Why it was created — connections are expensive

Imagine 1,000 users, each opening its own database connection. Every new connection requires a TCP handshake, authentication, authorization, memory allocation, and session creation — all of which consume resources. Opening and closing connections repeatedly is expensive, and RDS Proxy removes most of that overhead by pooling and reusing connections.

The Lambda + RDS problem

A classic case: an RDS MySQL instance allows a maximum of, say, 500 connections. Suddenly 10,000 Lambda functions try to connect. The result is “Too many connections” — the database starts rejecting requests and the application slows down or crashes.

Without proxy: 10,000 Lambdas → DB → “Too many connections” vs With proxy: Lambdas → RDS Proxy pools connections → DB stays healthy

When to use RDS Proxy

  • Using Lambda
  • Large number of short-lived connections
  • High-traffic applications
  • You need faster failover
  • You want IAM authentication
  • You want better database scalability

Faster failover

Normally, when the primary crashes the application loses its connections and must reconnect — which takes time. With the proxy in front, it holds the application connections and re-points them to the promoted instance, so failover is noticeably faster.

RDS Proxy is never public

The proxy is never publicly accessible: it cannot be reached from the internet, has no public IP, and allows no browser access. It must be accessed from within the VPC. A common serverless production architecture is:

Users↓ API Gateway↓ Lambda↓ RDS Proxy↓ Aurora / RDS (private)

RDS Proxy is itself serverless and autoscaling — AWS manages its capacity, scaling and availability, so you don't launch EC2 servers to run it.

6 · ElastiCache basics

ElastiCache is an in-memory cache used to reduce load on databases. By serving repeated reads from memory it dramatically cuts database queries.

Without cache: 1,000,000 DB queries / hour ↓ add cache After cache: 50,000 DB queries / hour

It is ideal for read-intensive workloads where reads far outnumber writes (Reads ≫ Writes), and it delivers very low latency (around 50 ms or less for cached reads). A typical layer order is EC2 → ElastiCache → RDS.

7 · Redis vs Memcached

Memcached

Memcached is essentially a simple Key → Value store, e.g. user:101 → Hariom, product:100 → iPhone. It's extremely simple (SET key value, GET key, DELETE key) and easy to learn — but that's basically all it does. It has no replication, no automatic failover, and no rich data types.

Redis — richer data types

Redis supports many data structures beyond plain key-value:

TypeExample
Stringuser:101 → Hariom
ListMessages: msg1, msg2, msg3
Hash (like a JSON object)User { name:"Hariom", age:22 }
Pub/Sub messagingPublisher → Redis → Subscribers (chat apps, notification systems)
Session storageStore user sessions centrally

Memcached cannot do Pub/Sub or these richer types.

Session storage & stateless applications

If a user logs in and the session is stored on Server 1, then a load balancer routes the next request to Server 2, the session is lost — a problem. The fix is to make the application stateless by storing sessions in Redis: the user hits the load balancer, which can send the request to any server, and every server reads the session from the shared Redis store.

8 · Caching strategies

⚠️

AWS does not automatically synchronize ElastiCache with RDS. Keeping the cache consistent is the application's responsibility — handled with patterns like Cache-Aside, Write-Through, TTL expiration, or event-driven invalidation. RDS stays the source of truth; Redis just holds temporary cached copies.

Scenario: data that never changes

If data never changes, caching is trivial — store it once and always serve it from cache. The hard part only appears when data does change (see cache invalidation below).

Lazy Loading / Cache-Aside (most common)

This is what most companies use. On a read, the application checks Redis first; on a miss it fetches from RDS and stores the result in Redis.

Request Product 100↓ Redis MISS↓ Fetch from RDS↓ Store in Redis

Cache entries are created only when users actually request them, which saves memory. The trade-off is that the first request for any item is slower (a guaranteed miss).

Write-Through

Instead of writing only to RDS, the application writes to the cache first (or to both), so the cache always holds the latest value.

Write Request↓ Update Cache (Redis)↓ Update Database (RDS)

Reads become almost always cache hits (very low latency); the cost is slower writes, since every write updates both stores.

Cache invalidation / stale data

This is one of the biggest challenges of caching. Example: an iPhone price of ₹50,000 is stored in both RDS and Redis. The seller changes it to ₹45,000 — RDS now has ₹45,000 but Redis still serves the old ₹50,000. To fix it the application deletes the stale key, e.g. await redis.del("product:100") — this is called cache invalidation. On the next request Redis misses, fetches ₹45,000 from RDS, and stores it again, so both are consistent. Large companies often automate this with events: when data changes, an event (via SNS/EventBridge) triggers a Lambda that clears the relevant Redis key.

Time-To-Live (TTL)

TTL sets an expiration time on each cached item; once the TTL elapses, the item is automatically removed and the next read becomes a miss that re-fetches fresh data from the database. It's a simple way to bound how stale cached data can get and to free memory, and it's commonly layered on top of Lazy Loading or Write-Through.

⏱️

TTL is about time-based expiry, not about which data gets loaded. So for "only load the data that's actually requested," the right strategy is Lazy Loading, not TTL; TTL just controls how long each entry lives once cached.

Cache evictions (memory pressure)

An eviction is different from an invalidation. Eviction happens when the cache runs out of memory and the engine forcibly removes items — often before they would naturally expire — to make room for new data. The symptom is a sudden rise in database hits because items keep disappearing from the cache.

📈

Fix evictions by adding capacity: scale up (a node type with more memory) or scale out (add more nodes/shards). Scaling down or in would make evictions worse. Evictions = memory problem; invalidation = staleness problem.

Aside: lazy loading in web development

The same idea appears in front-end work. Without lazy loading, <img src="large-image.jpg"> downloads immediately; with <img src="large-image.jpg" loading="lazy"> the browser loads it only when needed. Likewise a query like SELECT * FROM Users JOIN Orders JOIN Products loads everything up front even if the user never views orders — lazy loading fetches related data only when it's actually requested.

9 · ElastiCache high availability (Redis)

✅

For a highly available Redis cluster: create a Replication Group with one or more read replicas in different Availability Zones, and enable Multi-AZ with Automatic Failover.

Primary Redis · AZ-1↓ replication Replica Redis · AZ-2

If the primary fails, ElastiCache automatically promotes a replica to be the new primary and applications reconnect — keeping the cache available.

With cluster-mode disabled, a Redis shard can have up to 5 read replicas.

Configuration required

  • Enable a Replication Group
  • Add at least one read replica
  • Enable Multi-AZ
  • Enable Automatic Failover

Why the alternatives fall short

OptionWhy it isn't highly available
Single Redis nodeIf the one node fails, the cache is down.
MemcachedNo replication and no automatic failover.
Backups onlyBackups help recovery but give no automatic failover.

10 · Choosing a caching strategy

Question

A dynamic website is fronted by an ElastiCache cluster. Latency must be minimal for all read requests, and writes are allowed to take longer. Which caching strategy fits?

✅

Write-Through caching.

The clues: read latency is the top priority for all reads, and slower writes are acceptable. Write-Through updates both the cache and the database on every write, so data is already cached when users read it and nearly every read is a hit. Cache-Aside (lazy loading) wouldn't satisfy “minimum latency for all reads,” because the first request for any item is always a miss that has to hit the database. The downside of Write-Through — slower writes, since both the cache and database update on each write — is exactly what the requirement permits.

🧭

Quick recap of the core ideas: Read Replica scales reads (async, read-only) · Multi-AZ standby keeps you available (sync, no traffic) · Aurora shares one storage layer across 1 writer + up to 15 readers for fast failover · RDS Proxy pools connections (great with Lambda) · ElastiCache offloads reads · Cache-Aside is lazy and memory-cheap · Write-Through keeps reads fast at the cost of slower writes.
